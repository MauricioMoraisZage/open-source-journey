# Open Source Contribution Plan

This document describes the process I will follow before submitting my first
open-source contribution.

My objective is not only to open a pull request, but to understand the project,
follow its contribution standards, test my changes and respond correctly to
maintainer feedback.

## Project Priorities

My initial project priorities are:

1. cloud-init
2. Fastify
3. Docker Docs
4. Snapcraft
5. Node.js
6. Multipass
7. Charmcraft

The order may change as I learn more about each project and identify issues
that match my current skills.

## Project Selection Criteria

Before choosing a project for my first contribution, I will evaluate:

- Alignment with my Linux or backend experience
- Clarity of the contribution guide
- Local development requirements
- Availability of beginner-friendly issues
- Ability to reproduce the problem
- Time required to understand the relevant code
- Availability of tests and validation commands
- Whether another contributor is already working on the issue

## Phase 1 — Repository Research

For each selected project, I will:

1. Read the main README
2. Read the contribution guide
3. Identify the main programming languages
4. Understand the repository structure
5. Identify build and runtime requirements
6. Review recent merged pull requests
7. Review beginner-friendly open issues
8. Study the expected commit and pull request format

## Phase 2 — Local Environment

Before selecting an issue, I will try to:

1. Fork the repository
2. Clone my fork
3. Configure the upstream remote
4. Install the required dependencies
5. Build or run the project locally
6. Run the existing tests
7. Run formatting and linting checks
8. Document any setup problems and their solutions

## Phase 3 — Issue Selection

A suitable first issue should be:

- Small and clearly defined
- Reproducible in my local environment
- Related to code or documentation I can understand
- Not already assigned or being actively implemented
- Possible to test before opening a pull request

Before starting, I will read the full issue discussion and confirm whether the
work is still available.

## Phase 4 — Implementation

For each contribution, I will:

1. Create a dedicated Git branch
2. Make a small and focused change
3. Follow the project's coding and documentation standards
4. Add or update tests when required
5. Run the relevant validation commands
6. Review the final diff
7. Write a clear commit message

## Phase 5 — Pull Request

Before submitting a pull request, I will confirm that:

- The issue is referenced
- The change has a clear purpose
- Unrelated files were not modified
- Tests pass locally
- Formatting and linting checks pass
- The pull request description explains the change
- Testing evidence is included
- Maintainers are allowed to modify the branch when appropriate

## Phase 6 — Review Process

After submitting a pull request, I will:

1. Monitor automated checks
2. Read maintainer feedback carefully
3. Ask questions when feedback is unclear
4. Make requested corrections
5. Keep the pull request focused
6. Document what I learned from the review

## Definition of a Completed Contribution

A contribution will only be recorded as completed when:

- A pull request has been submitted to the original project
- The required checks have passed
- Maintainer feedback has been addressed
- The pull request has been merged

A pull request that is still under review will be recorded as:

> Contribution submitted and under review

## First Contribution Strategy

My first contribution does not need to be large.

A small documentation correction, test improvement, typing fix or focused bug
fix can help me learn the complete contribution workflow before working on
larger changes.
